# Upstream Etiquette

Field tests must respect upstream maintainers.

## Expectations

- Read the target repository contribution guide before public posting.
- Keep public comments focused on the user-visible issue, patch behavior, and
  validation.
- Do not pressure maintainers with Scarab-specific authority claims.
- Do not imply that SDS has authority over the upstream project.
- Separate diagnostic evidence from patch authorship.
- Accept maintainer feedback as project authority.

## Public PR Posture

A public PR should explain:

- the issue;
- the focused repair;
- why the change is narrow;
- validation run;
- limitations or non-goals.

Avoid language that sounds like automated patch farming or defensive proof.

## Optional Mascot Use

When a repository explicitly welcomes a friendly image, Scarab-owned public
communications may use the approved mascot image instead of unrelated cute
images. Keep it small, optional, and secondary to the technical evidence.

Do not use the mascot in security-sensitive reports, tense review threads, or
places where it could make the work feel less serious than the upstream issue.
